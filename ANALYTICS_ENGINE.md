# ANALYTICS & REPORTING ENGINE
## Rozi Khan Dropshipping Platform

**Document Version:** 1.0
**Author:** Senior Data Architect

---

## 1. System Overview
The Analytics Engine processes vast amounts of transactional data to provide actionable intelligence for Super Admins, Suppliers, and Retailers. It powers the UI dashboards with charts, generates scheduled email reports, and handles heavy aggregations without degrading the performance of the core transactional API.

---

## 2. Core Metrics & KPIs

### 2.1 Super Admin Metrics
* **Total GMV (Gross Merchandise Value):** Platform-wide revenue.
* **Platform Revenue:** Commission fees collected.
* **Active Entities:** Count of active Suppliers and Retailers.

### 2.2 Supplier Metrics
* **Total Wholesale Sales:** Revenue generated by their catalog.
* **Fulfillment SLA:** Average time taken from `PAID` to `SHIPPED`.
* **RMA / RTO Rate:** Percentage of items returned or undelivered.

### 2.3 Retailer Metrics
* **Gross Profit:** Retail Sales - (Wholesale Cost + Shipping).
* **Conversion Rate:** If utilizing platform storefront tracking.
* **Top Performing Products:** Highest volume imported products.

## 3. Database & Aggregation Strategy

Running complex `SUM()` and `GROUP BY` queries on millions of rows in the active `orders` table causes locks and degrades performance. 

### 3.1 Aggregation Strategy
We utilize an **ETL (Extract, Transform, Load)** pattern.
* **Transactional DB (OLTP):** The main PostgreSQL tables (`orders`, `payments`).
* **Analytical Views (OLAP):** PostgreSQL Materialized Views.

### 3.2 Materialized Views
A scheduled Celery cron job runs every night at 2:00 AM UTC to refresh materialized views:
```sql
CREATE MATERIALIZED VIEW mv_daily_supplier_revenue AS
SELECT supplier_id, DATE(created_at) as order_date, SUM(total_wholesale_amount) as revenue
FROM orders WHERE status IN ('DELIVERED', 'SHIPPED') GROUP BY supplier_id, DATE(created_at);
```

---

## 4. Caching Strategy (Redis)

Dashboard loading must be instantaneous (<200ms).
1. When a user requests their dashboard, FastAPI queries Redis: `GET supplier:dashboard:{id}:today`.
2. If cache miss, it queries the Materialized View, formats the JSON for Recharts/Chart.js, and saves it back to Redis with an expiration of 15 minutes.
3. Fast-moving stats (like 'Today's Orders') are incremented directly in Redis during the order routing workflow to provide real-time updates.

---

## 5. Export System & Scheduled Reports

* **Exports:** Users can request CSV/Excel exports. For large queries, FastAPI dispatches a Celery task `generate_export()`. The task queries the DB, writes the CSV to S3, and emails a presigned S3 download link to the user.
* **Scheduled Reports:** Celery beat schedules run weekly logic (e.g., Sunday 11 PM) to compile a "Weekly Performance Summary" and push it to the Notification Engine for email delivery via AWS SES.
